# Purchase Order — CW ↔ internal mapping

**CW contract:** `docs/Insurance REST API-v17-20260304_100318.pdf` §3.3.8 (Purchase Orders)
**Internal tables:** `purchase_orders`, `purchase_order_groups`, `purchase_order_combos`, `purchase_order_items` (see `apps/api/src/database/schema/index.ts` and `docs/design/01_DB_DESIGN.md`). Inline invoices are delegated to `CrunchworkInvoiceMapper` and land in `invoices` (see [`docs/mapping`](./README.md) once the invoice mapping doc is written).
**Mapper:** `apps/api/src/modules/external/mappers/crunchwork-purchase-order.mapper.ts`
**Last aligned with:** Insurance REST API v17 (exported 2026‑03‑04)

This document is the **spec** for the PO mapper. Where current code does not yet match a row below, §11 (_Known deviations / backlog_) calls it out explicitly so nothing is silently dropped.

---

## 1. Destination categories

Every CW field from §3.3.8 is routed to exactly one destination:

| Category | Target | Notes |
|---|---|---|
| Promoted column | explicit column on `purchase_orders` | Queryable, indexed where relevant |
| Lookup FK | `*_lookup_id` column | Resolved via `lookup_values.external_reference` within a named domain |
| External link / FK | `job_id`, `claim_id`, `vendor_id`, `quote_id` | Resolved through `external_links` / the external-object graph |
| JSONB bucket | `po_to`, `po_for`, `po_from`, `service_window`, `adjustment_info`, `allocation_context` | Queryable via JSONB operators |
| Child table | `purchase_order_groups` → `purchase_order_combos` → `purchase_order_items`; inline `invoices[]` → `invoices` (delegated) | One row per array element |
| `purchase_order_payload` | full CW response | Always stored; lossless fallback |

The `purchase_order_payload` column **always** contains the full verbatim CW response. Every field is therefore preserved losslessly even when it is also promoted to a column, lookup, or JSONB bucket below.

## 2. Identity, references, timestamps

| CW field | Type | Internal destination | Notes |
|---|---|---|---|
| `id` | String (UUID) | `external_links` (row keyed on `external_objects.provider_entity_id = id`) | CW's canonical PO UUID. There is **no** `external_reference` column on `purchase_orders`; identity lives in `external_links`. |
| `externalId` | String | `external_id` (column) | Insurer's own PO reference (e.g. `"ExternalId"`). Do **not** confuse with the CW `id` UUID. |
| `purchaseOrderNumber` | String | `purchase_order_number` (column) | Auto-generated by CW; read-only. |
| `tenantId` | String | _ignored on ingest_ | CW tenant is already known from the connection context; do **not** overwrite `purchase_orders.tenant_id`. |
| `createdAtDate` | String (ISO 8601) | `purchase_order_payload.createdAtDate` | Captured in payload; no promoted column. |
| `updatedAtDate` | String (ISO 8601) | `purchase_order_payload.updatedAtDate` | Captured in payload; no promoted column. |
| `createdBy.{id,name,externalReference}` | Object | `created_by_user_id` (column) ← `createdBy.externalReference`; remainder in `purchase_order_payload` | CW user reference. |
| `updatedBy.{id,name,externalReference}` | Object | `updated_by_user_id` (column) ← `updatedBy.externalReference`; remainder in `purchase_order_payload` | CW user reference. |

## 3. Parent / related entity resolution

Each of the fields below is resolved against `external_links` / `external_objects` for the same CW connection. Resolution failures are **soft**: the column is left null, the raw CW id remains in `purchase_order_payload`, and the mapper may re-resolve on a subsequent `purchase_order.*` event after the parent entity arrives.

| CW field | Internal column | Provider entity type | If unresolved |
|---|---|---|---|
| `jobId` | `job_id` | `job` | Leave null; log `UNRESOLVED_PARENT` if both `jobId` and `claimId` are unresolved. |
| `claimId` | `claim_id` | `claim` | Leave null. |
| `vendor.id` (primary) / `vendor.externalReference` (fallback) | `vendor_id` | `vendor` | Leave null; `vendor.name` remains in `purchase_order_payload.vendor.name`. CW rule for `vendor.externalReference`: "Continue API call" on unknown value. |
| `quoteRevisionId` | `quote_id` | `quote` | Leave null; raw id remains in `purchase_order_payload.quoteRevisionId`. |

Per the CW contract a PO must have **either** `jobId` **or** `claimId`; the DB enforces this via `chk_po_parent` (`claim_id IS NOT NULL OR job_id IS NOT NULL`). The mapper therefore skips projection with `{ skipped: 'skipped_no_parent' }` if **both** parents remain unresolved, so the orchestrator can retry after the parent event arrives.

## 4. Lookup references

Each CW lookup object has the shape `{ id, name, externalReference }`. We resolve `externalReference` against `lookup_values` within the named domain. If unresolved, behaviour matches the rule column in the CW contract.

| CW field | Internal column | Lookup domain | If unresolved |
|---|---|---|---|
| `status.externalReference` | `status_lookup_id` | `purchase_order_status` | Auto-create lookup stub (CW rule: "Continue API call"). |
| `purchaseOrderType.externalReference` | `purchase_order_type_lookup_id` | `purchase_order_type` | Auto-create lookup stub (CW rule says "Fail API call", but webhook ingress has no synchronous caller to reject; same rationale as `claims.md` §10 for `account`). Stub rows have `name === external_reference` and are logged to `external_reference_resolution_log` for back-fill. |

**Per-field `.id` and `.name`** from each lookup object are _not_ promoted to columns; they remain inside `purchase_order_payload`.

**Object-or-string tolerance.** When either `status` or `purchaseOrderType` arrives as a bare string, the mapper attempts a case-insensitive name match within the lookup domain. On miss the raw string is kept in `purchase_order_payload` and the FK is left null.

## 9. Child tables — line item hierarchy

The PO line-item tree (`groups → combos → items`) is rebuilt on every ingest using a **destroy-and-rebuild** strategy: existing `purchase_order_groups` for the PO are deleted (cascading to `purchase_order_combos` and `purchase_order_items` via FK), then the CW payload's tree is re-inserted. Catalog / quote / unit references are preserved verbatim in each row's JSONB payload so replay remains auditable.

### 9.1 `purchase_order_groups`

For each element in `payload.groups[]` insert one row, in the order given by `index` (stored in `sort_index`).

| CW field | Destination |

| `groups[].groupLabel.externalReference` | `group_label_lookup_id` via `group_label` lookup domain (auto-create per CW rule: "Fail API call" is relaxed to auto-create stub) |
| `groups[].groupLabel.{id,name}` | `group_payload.groupLabel` (verbatim) |
| `groups[].description` | `description` |
| `groups[].length` | `dimensions.length` (JSONB) |

### 9.4 Inline `invoices[]`

| CW field | Destination |
|---|---|
| `invoices[].id` | `external_links` (keyed on the invoice CW id) |
| `invoices[].purchaseOrderId` | `invoices.purchase_order_id` (this PO) |
| remainder | `invoices` table per the invoice mapping doc (TBD; fields captured verbatim in `invoices.invoice_payload` until the spec lands). |

The PO mapper does **not** prune invoices. An invoice that disappears from a later CW PO payload is left intact because invoices also arrive via their own `invoice.*` webhook stream.

## 10. `purchase_order_payload`

- Replay / reprocess tooling
- Diff debugging
- Any future field not yet promoted by a newer version of this mapping

## 11. Known deviations / backlog

The sections above are the **spec** (target state). As of the PO mapper's current implementation, the following rows are not yet fully honoured. Anything listed here remains queryable via `purchase_order_payload`.

| Spec section | Gap in current mapper | Tracking |
|---|---|---|
| §2 `externalId` | The column `external_id` is currently populated from the CW `id` UUID instead of CW's `externalId` scalar. `externalId` therefore survives only in `purchase_order_payload`. | Fix alongside vendor / quote resolution. |
| §3 `vendor_id`, `quote_id` | Neither FK is resolved; both columns stay null even when the CW payload includes `vendor` / `quoteRevisionId`. | Add `resolveFK` calls in the mapper. |
| §3 skip-on-no-parent | The mapper does not yet return `{ skipped: 'skipped_no_parent' }` when both `jobId` and `claimId` are unresolved; it inserts a row with both FKs null, which violates `chk_po_parent` and will raise a constraint error. | Mirror the quote mapper's skip path. |
| §5 `expiresInDays`, `service_window.*` duplicates | Only `start_date`/`end_date` are written; `start_time`/`end_time`/`expiresInDays` and the JSONB `service_window` bucket are not populated. | Extend `poData`. |
| §6 party buckets | `po_to` / `po_for` / `po_from` are read as objects (`payload.poTo` / `.poFor` / `.poFrom`) but CW sends **flat** fields (`toName`, `forEmail`, …). As a result the buckets and the `po_to_email` / `po_for_name` promoted columns are always empty. | Collect flat `to*`/`for*`/`from*` keys into the buckets. |
| §8 `adjustment_info`, `allocation_context` | Neither bucket is populated, even though the scalar columns `adjusted_total` / `adjusted_total_adjustment_amount` are. | Write the buckets alongside the scalars. |
| §7 `total_amount` | Mapper reads `payload.totalAmount` but the CW field is `total`. Column stays null. | Read `payload.total`. |
| §9.1 `group_label_lookup_id`, `dimensions`, `totals` | None of these are written on group rows today; `description` and `sort_index` are the only promoted fields. | Extend `syncLineItems`. |
| §9.1 group-level `items[]` | Only `groups[].combos[].items[]` are iterated; `groups[].items[]` (standalone items directly under a group) are dropped from `purchase_order_items`. They remain in `group_payload` for audit. | Iterate both arrays. |
| §9.2 `catalog_combo_id`, `quote_combo_id`, `totals` | Not written on combo rows. | Extend `syncLineItems`. |
| §9.3 `unit_type_lookup_id`, `catalog_item_id`, `quote_line_item_id`, `markup_type`, `markup_value`, `reconciliation`, `manual_allocation`, `tags`, `note` | Not written on item rows. Only `name`, `description`, `category`, `sub_category`, `item_type`, `quantity`, `tax`, `unit_cost`, `buy_cost`, `sort_index`, `item_payload` are populated. | Extend `syncLineItems`. |
| §9.4 inline invoices | Not forwarded to `CrunchworkInvoiceMapper` from the PO mapper. Invoices arrive only through their own webhook stream in the meantime. | Wire the delegation (see discussion doc `002c-entity-mappers.md` §3.3 step 7). |

Tracking each gap here keeps the mapping doc authoritative: every CW field from §3.3.8 has a declared home, and any that the mapper cannot yet write still survives in `purchase_order_payload`.

## 12. CW fields deliberately not mapped

| CW field | Reason |
|---|---|
| `id` on every nested object (`status.id`, `purchaseOrderType.id`, `vendor.id` beyond external-link resolution, `groupLabel.id`, `unitType.id`, `createdBy.id`, `updatedBy.id`, `groups[].id`, `combos[].id`, `items[].id`) | CW-internal identifiers; we key on `externalReference` (lookups) or the external-link graph (entities) instead. All preserved in `purchase_order_payload` / `group_payload` / `combo_payload` / `item_payload`. |
| `tenantId` (top-level and nested) | See §2 — tenant is derived from the connection. |
| `createdAtDate` / `updatedAtDate` / `createdBy` / `updatedBy` beyond the promoted user-id columns | Display metadata only; available in `purchase_order_payload`. |

Nothing else from §3.3.8 is unmapped.

## 13. Implementation notes

`CrunchworkPurchaseOrderMapper`
(`apps/api/src/modules/external/mappers/crunchwork-purchase-order.mapper.ts`)
projects CW PO payloads inside a single transaction supplied by
`InProcessProjectionService`. Behavioural notes that go beyond the table rows:

- **Existing-PO fallback order**: (1) link in `external_links` for the same
  `external_object`, (2) `(tenant_id, external_id)` once §2's `externalId`
  gap is closed, (3) `(tenant_id, purchase_order_number)`. Both fallbacks are
  backed by partial unique indexes on `purchase_orders`
  (`UQ_purchase_orders_tenant_external_id`, `UQ_purchase_orders_tenant_po_number`);
  until the middle fallback is implemented the mapper relies solely on the
  external-link row created on first ingest.
- **Line-item sync is destroy-and-rebuild.** On every ingest the mapper
  deletes all `purchase_order_groups` for the PO (cascading to combos and
  items) and re-inserts them. This keeps replay idempotent and avoids diffing
  a deeply nested tree. Catalog / quote / unit references are preserved in
  each row's `*_payload` JSONB for audit.
- **Lookup auto-create is opinionated.** `purchase_order_status`,
  `purchase_order_type`, `group_label`, and `unit_type` misses all auto-create
  a stub `lookup_values` row keyed on the external reference and log to
  `external_reference_resolution_log`. We diverge from the CW contract's "Fail
  API call" rule for `purchaseOrderType` for the same reason as `claims.md`
  §10 relaxes `account`: webhook ingress has no synchronous caller to reject,
  so stub creation is strictly preferable to dead-lettering the whole PO.
- **Tenant scoping is invariant.** CW's `tenantId` in the payload is never
  used to set `purchase_orders.tenant_id`; the mapper takes tenant from the
  `params.tenantId` supplied by the orchestrator, which was resolved from the
  connection context.
- **`purchase_order_payload` is always written verbatim.** Any CW field not
  handled by a row above — including fields added in a future CW API version —
  is preserved under `purchase_order_payload.<key>` for replay.
